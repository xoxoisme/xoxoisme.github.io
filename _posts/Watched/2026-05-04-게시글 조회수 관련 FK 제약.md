> "**컬렉션을 지워도 조회수 테이블의 기록이 그대로 남아있는 문제가 발생했다.**"

---

## 목표

> FK 제약에 대해 알아봅니다.

---

## 문제 상황

- 컬렉션을 삭제해도 해당 컬렉션의 조회 이력 로그가 DB에 남아 있는 고아 레코드 문제가 발생했다.
- 또한, `CollectionService.delete()`에서 `CollectionItem`은 삭제했으나 `CollectionView`는 삭제되지 않아 FK 제약이 생기면 삭제 시 무결성 오류가 발생할 수 있었다.

---

## 원인

```java
// 변경 전 — FK 없는 단순 Long 컬럼
@Column(nullable = false, name = "collection_id")
private Long collectionId;
```

- `collection_view` 테이블의 `collection_id`컬럼이 단순 `Long` 타입으로 선언되어 있다.

---

## 해결

### 1. `CollectionView` 엔티티 — `@ManyToOne` FK로 교체

```java
// 변경 후
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(nullable = false, name = "collection_id")
private Collection collection;
```

### 2. `CollectionViewRepository` — 삭제 메서드 추가

```java
void deleteByCollectionId(Long collectionId);
```

### 3. `CollectionService.delete()` — 뷰 로그 먼저 삭제

```java
// 삭제 순서: view → item → collection (FK 제약 위반 방지)
collectionViewRepository.deleteByCollectionId(collectionId);
collectionItemRepository.deleteByCollectionId(collectionId);
collectionRepository.delete(collection);
```

### 4. `CollectionView.create()` 호출부 수정

```java
// 변경 전
CollectionView.create(collectionId, userId)

// 변경 후
CollectionView.create(collection, userId)
```

- 이후, 컬렉션을 삭제하면 `collection_view`와 `CollectionItem`까지 함께 삭제되었다.
---

## User 는 FK제약 걸지 않은 이유

- 사실상, 조회 이력은 집계 데이터 성격으로, 고아 레코드가 남아도 서비스 동작에 영향이 없다.
- 다만, 아직 현재 Watched에는 회원 탈퇴 기능이 없어 구현 시 처리 예정이다.

## 싱글톤 패턴

> 생성자가 여러 번 호출되더라도 실제로 생성되는 객체는 하나이다.

- 하나의 클래스에 오직 하나의 인스턴스만 가진다.
- 보통 **DBCP(Database Connection Pool)**에서 많이 사용된다.
- 그 이유는 대용량의 DB를 사용할 때마다 연결하고 해제하는 건 큰 리소스 낭비가 되기 때문이다. 그래서 매니저 인스턴스를 하나만 만들어 관리하는 것이다.

```java
Class Singleton {
  private static class singleInstanceHolder {
    private static final Singleton INSTANCE = new Singleton();	// static으로 처음 한번만 생성한다.
  }
  public static Singleton getInstance() {
    return singleInstanceHolder.INSTANCE;
  }
}
```

- 위 예시는 여러 싱글톤 예제 중 **지연 초기화 홀더 클래스** 방식이다. 내부 Holder클래스를 활용해 초기화를 지연시켜두는 방식을 사용한다. 실제로 누가 해당 인스턴스를 만들기 전 메모리 낭비가 되지 않고, 처음 사용하게 되면 초기화가 진행되며 이후는 캐시된 단일 인스턴스를 반환한다.

### 장점

- 단일 인스턴스를 재사용하기에 메모리가 절약된다.
- 객체 로딩하는 시간이 처음 이후부터 빨라진다.
- 어디서나 동일한 인스턴스를 사용한다.

### 단점

- 어디서나 동일한 인스턴스를 가지기에 **강한 결합도**를 가지고 있다.
- TDD가 어려워진다.
  - `Mock` 주입 불가
  - 상태 초기화 어려움
- 동기화가 따로 없으면 동시성에 문제가 생길 가능성이 있다.
- 상속이 불가하다.

### DI시, 결합도를 느슨하게 할 수 있다.

> 상위 모듈은 하위 모듈에서 어떠한 것도 가져오지 않아야 한다.

- 의존성 주입(Dependecy Injection)을 통해 모듈 간의 결합을 느슨하게 만들 수 있다.
- 메인 모듈이 직접 하위 모듈에 의존성을 주입하지 않고, **중간 의존성 주입자**가 간접적으로 의존성을 주입하는 것을 의미한다. 이를 **디커플링**이라고도 한다.

```java
Class UserService {
  private MysqlUserRepository repository = new MysqlUserRepository();
  
  public void join(User user) {
    repository.save(user);
  }
}
```

- 위 예시는 직접 주입하는 방식이다.
- DB를 Mysql과 직접 연결하게 되면서 다른 DB를 연결 시 복잡성이 증가하게 된다.
- 단위테스트를 위한 `Mock` 주입이 불가해져 테스트가 힘들다.

```java
Class UserRepository implemets MysqlRepository{
  void save(User user);
}

Class UserService {
  private UserRepository repository;
  
  public UserService(UserRepository repository) {
    this.repository = repository;
  }
  
  public void join(User user) {
    repository.save(user);
  }
}
```

- UserRepository를 추상화함으로써 모듈 간의 관계를 느슨하게 할 수 있다.
- 이와 마찬가지로, TDD 또한 `Mock` 객체를 주입할 수 있어 테스트가 가능해진다.

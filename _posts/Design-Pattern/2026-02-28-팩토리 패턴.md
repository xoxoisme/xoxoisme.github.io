## 팩토리 패턴

> 상속을 핵심으로 활용하는 디자인 패턴이다.

- 상속 관계에 있는 두 클래스에서 상위 클래스가 중요한 뼈대를 결정하고, 하위 클래스에서 객체 생성에 관한 구체적인 내용을 결정한다.
- 상위/하위가 분리되기에 느슨한 결합을 가져 유연성을 가진다.

```java
// enum
public enum CoffeeType {

    LATTE("Latte"),
    AMERICANO("Americano");

    private final String value;

    CoffeeType(String value) {
        this.value = value;
    }

    public String getValue() {
        return value;
    }
}

// 추상 클래스
abstract class Coffee{
    public abstract String getName();
  }
}

// 상속 클래스(라떼)
public class Latte extends Coffee {
  
  @Override
  public String getName() {
    return CoffeeType.LATTE.getValue();
  }
}

// 상속 클래스(아메리카노)
public class Americano extends Coffee {
  
  @Override
  public String getName() {
    return CoffeeType.AMERICANO.getValue();
  }
}

// 팩토리 클래스 
public abstract class CoffeeFactory {

    public static Coffee createCoffee(CoffeeType type) {
        switch (type) {
            case LATTE:
                return new Latte();
            case AMERICANO:
                return new Americano();
            default:
                throw new IllegalArgumentException("Unknown Coffee Type");
        }
    }
}

```

- 위 예시는 객체 생성을 직접 하지 않고, **팩토리 클래스에게 맡겨서 원하는 객체를 생성**하는 구조이며 Simple Factory이다.
- 추상 클래스인 `Coffe`는 실제 존재하지 않고 개념일 뿐, 실존하는 건 `Latte`와 `Americano`가 된다.
- 다음, 상속 클래스는 오버라이딩을 통해 `Coffe`의 메서드를 재정의한다.
- 팩토리 클래스는 요청에 맞는 객체를 생성한다.

### 장점

- 객체 생성과 비즈니스 로직이 분리되어 유지보수가 용이해진다.
- 기존 코드를 변경하지 않고 새로운 객체 타입을 추가하거나 변경이 가능해 확장성이 좋다.
- 동일한 패턴을 재사용할 수 있다.

### 단점

- 새로운 객체마다 새로운 팩토리 클래스가 필요할 가능성이 있어 시스템의 복잡성이 증가한다.
- 클라이언트 코드가 팩토리 인터페이스에 의존하는 게 커져 인터페이스 설계가 중요해진다.

## 분산 환경이 만들어진 이유

- 토스 뱅크는 **초기에 다른 은행의 솔루션을 구매해 시스템을 구축**했고, 이는 거대한 단일 서버가 단일 DB를 바라보는 **모놀리식** 구조였다.
- 이로 인해 환전은 서버가 분리된 원화 계좌와 외화 계좌 위에서 구현되어야 했다.
- 단일 서버라면 트랜잭션으로 쉽게 원자성 보장이 되지만, 서로 다른 DB를 보는 분리된 서버에서는 실패에 대비해 보상 트랜잭션이 필요하다.

---

## 2PC vs SAGA

#### 2PC(Two-Phase Commit)

1. Prepare 단계: Coordinator가 참여자들에게 **커밋 가능 여부를 물어보고** 가능하면 각자 바라보고 있는 DB에 락을 걸어 대기한다.
2. Commit 단계: 이후 모두가 YES 응답이 오면 Coordinator가 커밋 명령하고, 하나라도 NO가 오면 롤백을 명령한다.

#### SAGA

- **각 서비스가 작은 로컬 트랜잭션을 실행하며 진행**하다가, 특정 단계에서 실패하면 이전에 커밋된 트랜잭션들에 **보상 트랜잭션**을 보내 롤백하는 방식이다.

| 구분 | 2PC | SAGA |
|------|-----|------|
| 동작 | 모든 참여자가 트랜잭션을 열고 **가장 느린 참여자의 투표까지 대기** | 각 서비스가 **로컬 트랜잭션**을 진행 |
| 가용성·확장성 | **낮음** | **높음** |
| 단점 | - 대기로 인해 전체 처리량 저하<br>- 네트워크 왕복이 많아 느림<br>- Coordinator 장애 시 모두 락을 물고 복구 대기 | - 일부만 커밋된 **중간 상태가 노출**될 수 있음<br>- 롤백을 위한 **보상 트랜잭션을 직접 구현**해야 함 |

→ 환전 서비스는 높은 트래픽 견뎌야하고 카드·회계 등 참여자 추가할 수 있어 **SAGA 패턴** 선택했다.

#### SAGA 2종류

| 구분 | 코레오그래피 (Choreography) | 오케스트레이션 (Orchestration) |
|------|---------------------------|-------------------------------|
| 방식 | 중앙 제어자 없이 메시지 **브로커로 이벤트 교환** | **오케스트레이터**가 각 서비스에 트랜잭션·보상 트랜잭션을 명령 |
| 장점 | - 단일 장애점 없음<br>- 서비스 간 **느슨한 결합** | 진행 중인 **상태 추적이 쉬움** |
| 단점 | 진행 상태 추적·디버깅이 어려움 | 오케스트레이터가 **단일 장애점**이 되고 모든 서비스와 결합 |

→ 토스는 **오케스트레이션 사가**를 선택했다

1. 클라이언트 요청을 받아 환전을 시작하는 환전 서버가 필요했다.
2. 진행 중인 환전 상태를 관리해야 했다.(한 사용자가 동시에 환전 요청 3건 보내면 각 요청이 진행 중인지, 완료인지 확인해야 함)

## 환전 구현

#### 성공 케이스

1. 환전 서버 → 원화 계좌에 **1,300원 출금** 요청
2. 출금 성공 → 외화 계좌에 **1달러 입금** 요청
3. 입금까지 성공 → 환전 성공 종료

#### 실패 케이스

- 정상적인 실패
    - 잔액 부족
    - 계좌 해지
- 비정상적인 실패
    - **서버/네트워크 문제**
    - 타임아웃

#### 정상적인 실패 처리

- 출금이 실패하면, 간단하게 추가 입금 요청없이 실패 처리하면 된다.
- 입금이 실패하면, 이미 원화 계좌에서 출금이 된 상태이기에 외화 계좌에 보상 트랜잭션이 필요하다.

#### 여기서 중요한 건, **환전 시 항상 출금**부터 해야한다는 점이다.

- 만약 입금을 먼저 진행하면, 외화 계좌에 1달러가 들어가게 되었다고 가정하자.
- 사용자의 출금 계좌에서 1300원을 요청했는데 실패했다면, 이 입금한 1달러를 다시 찾아와야 하는데 그 사이에 카드 결제 요청으로 인해 1달러를 사용했다면 되돌릴 수가 없다.

#### 통신 방식

- **입금과 출금은 HTTP**를 사용해 동기적으로 처리했다.
    - 출금 결과를 알고 입금으로 넘어가야 한다.
    - 사용자가 환전이 즉시 완료되기를 기대하고, 지연이 되면 이를 위해 **타임아웃이 필요**했다.(비동기는 요청과 응답이 시간적으로 분리되어 있기에 주기적으로 폴링하기엔 리소스낭비 → try-catch로 간단 해결가능)
- 보상 트랜잭션은 메시징을 이용하여 비동기적으로 처리했다.
    - 출금 취소는 유저 입장에서는 백그라운드에서 처리되면 충분하다.
    - 입금이 실패하면, 출금된 돈은 반드시 유저에게 돌려줘야하기에 **에러 핸들링을 메세지 브로커와 원화 계좌 컨슈머에 위임해 정합성을 보장**하기 유리하다.(브로커가 출금 취소 메세지를 보관하기에 컨슈머가 죽어도 재시도를 자동으로 반복)

## 비정상적인 에러 핸들링

- **타임아웃의 경우** 실제로 실패했거나, 혹은 동작은 했는데 지연처리 되었거나 중 하나이다.
- 그래서 출금 결과를 다시 확인해 후속 처리한다.
- 이는, 입금도 마찬가지이다.

```
출금 성공 → 보상(입금) 처리
출금 실패 → 추가 조치 없이 환전 실패 처리

입금 성공 → 출금·입금 모두 성공이므로 환전 성공 처리
입금 실패 → 앞서 성공한 출금에 출금 취소를 보내 환전 롤백
```

#### 다만, 여기서 결과를 확인할려고 하는 거 조차 실패할 확률이 높다.(상대 서버/네트워크 문제라면)

- 일반적으로, Producer가 메세지 발행하면 즉시 Broker를 통해 Consumer에게 전달된다.
- 다만, 지연 시간을 넣어 발행하면 별도 지연 토픽으로 Kafaka 메세지 스케줄러에 전달된다.
- 스케줄러는 지연 시간만큼 기다린 후 원래 토 픽의 메세지를 대신 발행해 Consumer가 지연 시간 이후 가져가게 한다.
- **Producer와 Consumer를 모두 환전 서버로 두면, 알림을 예약하는 효과**를 줄 수 있고 상대 서버에 회복할 시간을 줄 수 있다.

→ 위 방법으로 **지연시간 딜레이**를 주고 정해진 횟수가 모두 초과되면 수동으로 메세지 발행하도록 했다.

#### 여기서 또! 환전 서버 자체가 죽었을 경우 배치 재처리로 해결한다.

- 지연 메세지 조차 발행하지 못하고 서버가 죽으면, Kafka에 아무 메세지도 없으니 아무도 재처리를 할 수 없다.
- 배치로 복구 가능한 이유는 오케스트레이션 SAGA의 특성 때문이다.
    - 환전 서버(**오케스트레이터**)는 모든 환전의 **현재 상태를 자신의 DB에 저장**하고 있다.

```
환전 DB:
┌──────────┬────────────────┬──────────┐
│ 환전 ID  │     상태       │  시각    │
├──────────┼────────────────┼──────────┤
│ TX-001   │ 출금완료/입금중 │ 10:30:01 │  ← 여기서 서버 죽음
│ TX-002   │ 완료           │ 10:29:55 │
└──────────┴────────────────┴──────────┘
```

- 중단된 상태부터 재시작이 가능하다.(출금까지 성공한 상태라면 출금 취소만 하면 됨)

#### 출금 취소 메세지 처리 중 에러

- 입금 실패 시 출금된 돈은 꼭 돌려줘야 하는데, 원화 계좌가 출금 취소를 처리하다 에러가 났을 경우 토스뱅크는 Consumer 데드레터(DL)로 카프카 메세지의 결과적 정합성을 보장한다.
- 지연 메세지 발행을 딜레이 별로 해도 실패하면, DL 서버를 통해 다시 발행한다.
- Consumer DL 서버에 메세지가 쌓이면 사람이 개입하여 해결한다.(보통, 코드 버그/데이터가 잘못됨/예상치 못한 비즈니스 규칙 위반으로 인함)

#### 메세지 발행 자체가 안됨

- Producer 데드레터(DL)를 사용한다.
- Producer DL 서버에 메세지를 대신 발행하고, DL 서버는 일정 시간 이후 회복된 서비스 메세지 브로커로 메세지를 다시 전달한다.

→ Consumer/Producer **어디서든 실패하면 DL이 받아서 보관하고, 결국엔 반드시 처리되도록 보장**한다.

## 모니터링

- **스테이트 머신과 이벤트 소싱 방식 저장**으로 진행된다.
- 토스 뱅크는 덮어쓰지 않고 추가 삽입(INSERT) 방식을 사용한다.(환전이 어디서 실패했는지 추적할 수 있음)
- 멈춘 환전을 탐지할 수 있다.(모든 환전 상태 관리하기에)
- 입출금 쌍으로 검증할 수 있다.(ETL로 배치가 쌍을 매칭해서 검증)

#### ETL
- **E**xtract(추출)·**T**ransform(변환)·**L**oad(적재) 의 약자이다.
- 두 DB를 직접 조회해서 비교하면 서로 다른 시스템이기에 어렵다.
- 그래서, ETL로 두 시스템의 데이터를 주기적으로 꺼내서, **분석하기 좋게 가공한 뒤, 하나의 분석용 DB에 저장**한다.

## 목표

> 트랜잭션 격리 수준에서 각 레벨의 특징에 대해서 알아봅니다.

---

## ACID 원칙

|        이름         |             설명              |
| :-----------------: | :---------------------------: |
|  Atomicity(원자성)  | 성공은 영구저장/실패는 되돌림 |
| Consistency(일관성) |        항상 정상 유지         |
|  Isolation(고립성)  |           간섭 금지           |
| Durability(지속성)  |           영구 보존           |

- ACID 원칙 중 트랜잭션 간 간섭하지 못하도록 하는 **Isolation(고립성)**과 관련해 격리 수준에 대한 레벨이 존재한다.

---

## 고립성 격리 레벨

| 레벨 |       이름       |    발생가능 문제    |            영향             |
| :--: | :--------------: | :-----------------: | :-------------------------: |
|  0   | Read Uncommitted |     Dirty Read      | 커밋되지 않은 데이터도 읽음 |
|  1   |  Read Committed  | Non-repeatable Read |    커밋된 데이터만 읽음     |
|  2   | Repeatable Read  |    PHANTOM READ     | 조회는 고정되나 삽입은 허용 |
|  3   | Seralizable Read |        없음         |   성능 저하/병렬처리 제한   |

### Read Uncommitted(0)

> 성능을 최우선으로 하나, 실무에선 거의 쓰지 않는다.

- 격리 수준을 아예 설정하지 않은 것이다.
- `COMMIT`/`ROLLBACK` 하지 않은 데이터도 접근이 가능하다.

**Dirty Read(확정되지 않은 데이터를 조회) 문제가 발생할 가능성이 있다.**

```sql
INSERT INTO account VALUES (1, 1000);

-- 1.트랜잭션 A가 가격을 800으로 변경한다.
UPDATE account SET balance = 800 WHERE id = 1;

-- 2.트랜잭션 B가 가격을 조회한다.(balance = 800)
SELECT balance FROM account WHERE id = 1;

-- 3.트랜잭션 A가 1을 롤백한다.(balance = 1000)
ROLLBACK;
```

- 트랜잭션 B가 읽은 가격(800)은 의미 없는 데이터가 된다.

### Read Commited(1)

> 조회를 제외한, 삽입/삭제/수정은 락 영향을 받는다.

- `COMMIT`한 데이터를 다른 트랜잭션이 접근하게 한다.
  - 조회는 `COMMIT` 상관없이 가능하다.

**Non-repeatable Read(비반복 읽기) 문제가 발생할 가능성이 있다.**

```sql
INSERT INTO account VALUES (1, 1000);

-- 1.트랜잭션 B가 가격을 조회한다.(balance = 1000)
SELECT balance FROM account WHERE id = 1;

-- 2.트랜잭션 A가 가격을 800으로 변경 후 커밋한다.
UPDATE account SET balance = 800 WHERE id = 1;
COMMIT;

-- 3.트랜잭션 B가 가격을 조회한다.(balance = 800)
SELECT balance FROM account WHERE id = 1;
```

- 트랜잭션 B는 같은 행을 두 번 조회했으나, 값이 달라진다.

### Repeatable Read(2)

> 동일한 행에 대한 조회를 막지만, 새로운 행의 삽입은 막지 못한다.

- 조회한 행에 대한 락을 걸 수 있다.

**Phantom Read(유령 읽기) 문제가 발생할 가능성이 있다.**

```sql
INSERT INTO account VALUES (1, 1000);

-- 1.트랜잭션 B가 범위 조회를 한다.(id = 1)
SELECT * FROM orders WHERE price >= 1000;

-- 2. 트랜잭션 A가 새로운 행을 추가한다.
INSERT INTO orders (id, price) VALUES (2, 1200);
COMMIT;

-- 3. 트랜잭션 B가 가격 범위 조회를 한다.(id = 1, 2)
SELECT * FROM orders WHERE price >= 1000;
```

- 트랜잭션 B는 같은 조건을 조회했으나, 새로운 데이터가 생겼다.

### Serializable Read(3)

> 가장 강력한 격리 수준이나, 성능과 동시성은 낮아진다.

- 트랜잭션 간 철저한 일관성을 유지한다.
- 이전 하위 레벨에서 발생한 모든 문제를 방지하지만, 성능면에서 좋지 않다.

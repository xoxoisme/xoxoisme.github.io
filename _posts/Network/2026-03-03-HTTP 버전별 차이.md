## 목표

> HTTP/1.0부터 HTTP/3까지 각각의 특징에 대해서 알아봅니다.

---

## HTTP/1.0

> 한 연결당 하나의 요청을 처리한다.

- 이로 인해 RTT(패킷의 총 왕복 시간)가 증가하는 단점이 있었다.

### RTT가 증가해 서버에 부담이 많이 가고 사용자 응답 시간이 길어졌다.

- 나름 여러 해소법이 존재한다.
  - **이미지 스플리팅**
    - 하나의 큰 이미지를 여러 개의 작은 이미지 조각으로 나누어 사용하는 기법이다.
    - 한 화면의 큰 이미지를 여러 이미지를 이어 붙였다 생각하면 된다.
  - **코드 압축**
    - 개행 문자, 빈칸을 없애 크기를 최소화하는 방법이다.
  - **Base64 인코딩**
    - 이미지 파일을 64진법으로 이루어진 문자열로 인코딩해 문서 안에 직접 포함하는 기법이다.
    - HTTP 요청은 줄어드나, 파일 크기가 증가하는 단점이 있다.

---

## HTTP/1.1

> 이전처럼 매번 TCP 연결을 하는 게 아닌, 옵션을 두어 연결을 끊기 전까지 지속 연결이 가능하게 했다.

- HTTP/1.0에도 `keep-alive`라는 옵션이 존재했지만, HTTP/1.1부터 표준화 되었다.

### 이전 패킷에 의해 지연될 때 성능이 저하하는 HOL Blocking 문제가 발생했다.

- 또한, 헤더구조에 많은 메타데이터가 들어있고 압축이 되지 않아 무거웠다.

---

## HTTP/2

> 하나의 TCP 연결에서 멀티플렉싱, 헤더 압축, 서버 푸시 같은 기능을 지원한다.

### 여러 개의 스트림을 사용하여 송수신하는 멀티플렉싱이 추가되었다.

- 특정 스트림의 패킷이 손실되더라도 해당 스트림에만 영향을 미치고 나머지 스트림은 정상 작동하게 했다.
- 애플리케이션 레벵의 HOL Blocking 문제는 해결했다.

### 다만, TCP 레벨의 HOL Blocking 문제는 여전히 존재한다.

- 요청 간 대기 문제는 해결했지만, 여전히 패킷 하나가 손실되면 그 뒤 데이터는 계속 대기한다.

### 허프만 코딩 압축 알고리즘을 통해 헤더를 압축 시킨다.

- 문자열을 문자 단위로 쪼개 빈도수를 세어 빈도가 큰 건 적은 비트 수를 빈도가 작으면 큰 비트 수를 사용해 필요한 비트 양을 줄인다.
- 가중치에 따라 줄이기에 확실히 줄어들 수 밖에 없다.

### 클라이언트 요청 없이 서버가 바로 리소스를 푸시할 수 있다.

---

## HTTPS

> TLS/SSL로 암호화해 전송함으로써 도청이나 변조를 방지하는 안전한 웹 통신 프로테콜이다.

- HTTP/2는 HTTPS 위에서 동작한다.

### 보안 세션을 기반으로 데이터를 암호화한다.

- 보안 세션이 만들어질 때 인증 메커니즘, 키 교환 암호화 알고리즘, 해싱 알고리즘이 사용된다.

### HTTPS를 구축하는 방법은 세가지가 존재한다.

- 직접 CA에서 구매한 인증키를 기반으로 서비스를 구착한다.
- 서버 앞단에 HTTPS를 제공하는 로드밸런서를 도입한다.
- 혹은, 서버 앞단에 HTTPS를 제공하는 CDN을 둔다.

---

## HTTP/3

> 현재 최신 버전이며, 이전 버전의 문제들을 해결하고 기본적으로 TLS 암호화를 제공하는 프로토콜이다.

- QUIC(UDP 기반) 위에서 동작한다.
- TCP의 3-way handshake를 진행하지 않기에 초기 연결 설정 시 지연 시간이 감소되는 장점이 있다.
- 열악한 네트워크 환경에서도 낮은 패킷 손실률을 자랑한다.

### 순서 제어는 두가지 방법이 존재한다.

- UDP 기반이기에 시퀀스가 아닌 **패킷넘버**를 붙인다.
  - 재전송해도 번호 재사용을 하지 않기에 다음 패킷이 대기하지 않는다.
  - 건너 뛰는 번호가 있으면 손실을 감지한다.
- QUIC의 핵심인 **스트림 오프셋**이 존재한다.
  - 도착한 프레임을 오프셋 기준으로 버퍼에 저장한다. 이후, 빠진 구간이 있으면 해당 스트림만 대기한다.
  - 연속 구간이 완성되면 그 때 애플리케이션에 전달한다.

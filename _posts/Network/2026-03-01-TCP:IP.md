## TCP

> 연결 지향, 신뢰성, 흐름 및 혼잡 제어를 하는 네트워크 프로토콜이다.

- **Trasmission Contorl Protocol(전송 제어 프로토콜)**은 인터넷에서 신뢰성있게 데이터를 주고받기 위해 사용하는 전송 계층 프로토콜이다.
- 데이터를 주고 받기 전 송신자와 수신자를 연결하는 역할을 하고 데이터를 다 주고 받았으면 끊는 역할도 한다.
- 세그먼트에 시퀀스 번호를 붙이고, 수신자 쪽이 ACK로 응답하며 데이터 유실이 되면 재전송한다.

### 3-way handshake

> TCP 프로토콜로 통신하기 전 상호연결을 수립하는 과정이다.

<img src="/images/2026-03-01-TCP:IP/스크린샷 2026-02-14 오후 2.38.47.png" alt="tcp/ip">

- 시퀀스 번호를 **동기화**한다.
  - 클라이언트와 서버는 서로 독립된 네트워크이다. 초기 시퀀스 번호가 다를 수 있다.
- 시퀀스 번호는 **난수**이다.
  - 이전 연결과 다른 완전히 새로운 연결임을 알리기 위함이다.
  - 시퀀스 번호를 예측 불가하게 하여 중간에 세션 하이재킹과 같은 공격을 막기 위함이다.
- 이전 시퀀스에 1을 더하는 이유는 **패킷의 시퀀스를 바이트 단위로 취급**했기 때문이다.
  - `SYN` 패킷은 안에 데이터가 없지만 신호용으로 1바이트 취급을 하고 연속된 데이터 스트림을 보장하기에 순서를 제어하기 위해 번호가 있는 것이다.

### 끊을 때는 4-way handshake인 이유는 TCP가 Full-Duplex(양방향 독립 전송)프로토콜 이기 때문이다.

> 한쪽이 데이터 전송을 끝냈다고 해서, 다른 쪽이 끝난 것은 아니다.

- 연결할 때 처럼 `SYN` + `ACK`를 함께 전송할 수 없다는 게 핵심이다.
- 현재 한 쪽은 데이터를 전송했다고 하더라도 다른 쪽은 데이터를 받는 중일 수 있기에 신뢰성을 기반으로 하는 TCP 프로토콜 특성 상 바로 종료할 수 없다.
- 그래서 데이터 전송이 양 쪽 모두 완료된 후, `FIN` 패킷을 보내는 과정이 추가되어 4-way인 것이다.

### 흐름

> 커널 영역 복사 → 세그먼트 생성 → 패킷 감싸기 → 프레임 감싸기 → 네트워크 전송

#### 애플리케이션 계층

- 사용자가 요청 생성을 한다.
- OS 커널에 해당 데이터를 **복사**한다.
  - 복사하는 이유는 사용자 영역은 언제든지 내용이 변경 가능하기에 신뢰성 붕괴를 미리 방지하기 위해 참조하지 않고 복사하는 것이다.

#### 전송 계층

- 데이터가 TCP로 전송된다.
  - 포트 번호 추가(출발지/목적지)
  - 시퀀스 번호 부여
  - 흐름 제어/혼잡 제어
- TCP **세그먼트 형태로 변환**된다.
  - 데이터를 버퍼의 큰 덩어리로 보내게 된다면 일부가 유실되면 처음부터 다시 재전송해야하고 순서 관리가 어렵다. 그래서 세그먼트로 나누어 유실되더라도 일부만 재전송하면 되기에 네트워크 부담이 적어진다.

#### 네트워크 계층

- 세그먼트가 IP 계층으로 전달된다.
  - IP 추가(출발지/목적지)
  - 라우팅 경로 설정
- **패킷**에 감싸진다.
  - IP 헤더 정보를 추가해야하기 때문이다.

#### 데이터 링크 계층

- **프레임**에 다시 감싸진다.
  - 목적지 MAC 주소 확인
  - 이더넷 헤더 추가

#### 물리 계층

- 프레임이 **전기 신호/광 신호/무선 신호**로 변환되어 실제 네트워크를 통해 전송된다.

### TCP는 어떻게 보낼지를 담당하고, IP는 어디로 보낼지를 담당한다.

- 네트워크 계층에서 IP는 목적지 IP를 정할 뿐 데이터가 잘 도착했는지, 순서대로 갔는지, 중복인지를 확인하지 않는다.
- 그저 라우팅을 통해 데이터를 목적지 네트워크까지 전달하는 역할만 수행할 뿐이다.

### 그러면 IP가 TCP가 하는 역할까지 같이 해주면 안되는 것일까?

- 네트워크 설계 철학은 **계층 분리**이다.
- IP가 신뢰성까지 담당하면, 라우터가 복잡해지고, 확장성이 매우 떨어져 인터넷 규모 확장이 어려워진다.

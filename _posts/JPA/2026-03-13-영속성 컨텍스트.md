## 영속성 컨텍스트

- 엔티티를 **영구 저장하는 환경**이라는 뜻이다.
- Entity Manager(엔티티 매니저)로 엔티티를 저장하거나 조회하면 영속성 컨텍스트에 보관하고 관리할 수 있다.

```java
em.persist(entity);           // entity를 영속성 컨텍스트에 저장(영속 상태로 만듬)
em.find(Entity.class, pk);    // entity를 DB에서 로드(영속 상태로 만듬)
```

## 엔티티 생명 주기

- 비영속(New/Transient) : 영속성 컨텍스트와 전혀 관계가 없는 **새로운 상태**
- 영속(Managed) : 영속성 컨텍스트에 **관리되는 상태**
- 준영속(Detached) : 영속성 컨텍스트에 저장되었다가 **분리된 상태**
- 삭제(Removed) : **삭제된 상태**

### 비영속

```java
User user = new User();
user.setId(1L);
User.setName("홍길동");
```

- 엔티티 **객체를 생성만 한 상태**이다.
- 아직 `User` 객체는 DB에 로드 되지 않고, 생성만 된 상태임을 알 수 있다.

### 영속

```java
em.persist(User);
em.find(User.class, id);
```

- 엔티티 매니저를 통해 영속성 컨텍스트에 저장하거나, DB에서 조회하면 영속성 컨텍스트가 엔티티를 관리한다.
- `@Id`가 붙은 필드로 테이블의 **기본키와 매핑한 식별자 값만 찾을 수 있다.**

#### 영속성 컨텍스트의 1차 캐시 구조는 `Map<EntityKey, Entity>`로 되어 있다.

- 그래서, 식별자 값만 찾을 수 있는 것이다.

### 준영속

```java
em.detach(user);		// 회원 엔티티를 영속성 컨텍스트에서 분리
em.clear();         // 영속성 컨텍스트 초기화
em.close();         // 엔티티 매니저(영속성 컨텍스트) 종료
```

- 영속성 컨텍스트가 관리하지 않게 되었을 상태를 말한다.

### 삭제

```java
em.remove(user);
```

- 엔티티를 영속성 컨텍스트와 DB에서 `flush()` 호출 시 삭제한다.

## 영속성 컨텍스트 쓰는 이유

### 1차 캐시

- 영속성 컨텍스트는 내부에 엔티티를 저장할 수 있는 **1차 캐시가 존재한다.**
- 엔티티를 처음 저장하면 1차 캐시에 저장된다.

#### 1차 캐시에 조회하려는 엔티티가 있으면 DB를 거치지 않고 메모리 상에서 바로 조회할 수 있는 것이다.

### 동일성 보장

```java
User a = em.find(User.class, "1L");
User b = em.find(User.class, "1L");
```

- `a == b`를 하면 `true`가 나오는데, 이건 **참조하는 주소값이 같다**는 뜻이다.
- 1차 캐시에 있는 동일한 인스턴스를 반환하고 있기 때문이다.
- DB를 거쳐서 새로운 인스턴스를 생성한다면, 동일성이 보장되지 않았을 것이다.

### 트랜잭션을 지원하는 쓰기 지연

```java
EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();
tx.begin();

em.persist(userA);
em.persist(userB);

tx.commit();
```

- 엔티티 매니저는 `commit()`이 호출되기 전까지 내부 쿼리에 실행된 메서드에 해당하는 SQL들을 저장소에 모아둔다.
- `commit()` 호출이 되면, DB에 SQL을 날려 DB에 반영된다.
- 이를 통해, 모아둔 쿼리를 DB에 한번에 날려, 매번 DB를 왕복하지 않게하며 성능적으로 이점을 보는 것이다.

### 변경 감지(Dirty Checking)

- JPA에서는 엔티티를 수정 시 `update()`가 필요 없다. 영속성 컨텍스트 안의 엔티티를 수정하면, **엔티티의 변경사항을 자동으로 DB에 반영**해준다.
- JPA는 영속성 컨텍스트에 보관할 때, **스냅샷을 한번만 저장**한다. 그리고 `flush()`가 호출되면 스냅샷과 엔티티를 변경해서 변경된 부분을 자동으로 반영한다.

#### 여기서 중요한 건, 더티 체킹은 필드의 모든 값에 대한 `update()`를 호출한다.

- 필드 하나만 변경되더라도 `update`는 모든 필드를 변경한다.
- 필드가 20개가 넘으면 성능 저하가 발생할 수 있어 `@DynamicUpdate`를 사용해 수정된 필드만 `update`를 날릴 수 있다.

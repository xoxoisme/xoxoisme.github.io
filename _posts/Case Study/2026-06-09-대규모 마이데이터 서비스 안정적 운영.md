## 마이데이터 서비스

- 여러 금융 기관에 흩어진 개인 신용정보를 한 곳에서 조회하게 해주는 서비스이다.
- 7만 TPS가 발생한다.

---

## 장애 대응 - Coordinator 시스템

- 기관에서 불러올 때 기관 장애가 마이데이터 서버에 이어 다른 토스 시스템으로 쉽게 전파되었다.
- 처음엔 오픈소스 Resilience4j(리실리언스포제이)로 기관별 서킷 브레이커를 적용했지만, **서킷 상태가 서버 단위로 관리된다는 한계**가 있었다.
- 이로 인해, 표본을 수집 못한 서버는 열리지 않아 시스템 동작이 일관되지 못하는 문제가 발생했다.
- 그래서 운영 자동화 기능을 분리한 별도 서버(Coordinator)를 제작했다.

#### 서킷 브레이커

- 외부 API가 문제가 생겼을 때 우리 서버가 다운 될 수 있기에 실패율이 임계치를 넘으면 자동으로 차단(Open)해서 내 서비스를 보호하는 역할을 한다.

```text
사용자 요청
    ↓
로드밸런서
  ↙  ↓  ↘
 A   B   C   ← 각자 독립적으로 서킷 상태 관리
 🔴  🟢  🟢
(차단) (통과)(통과)
    ↓    ↓   ↓
      외부 기관 API (고장난 상태)
```

- 위처럼 고장이 난 상태의 외부 기관 API에 그대로 요청이 들어가 스레드 블로킹이 된다.(해당 서버는 임계치가 넘지 않았기에)

- 이로 인해, 표본을 수집 못한 서버는 열리지 않아 시스템 동작이 일관되지 못하는 문제가 발생했다.
- 그래서 운영 자동화 기능을 분리한 별도 서버(Coordinator)를 제작했다.

#### 동작 방식

1. 마이데이터 서버가 배포 시 Coordinator에 등록(서버 ID 발급, 이후 하트비트로 상태관리)
2. 마이데이터 서버는 성공/실패를 카운트 해 버퍼에 모았다가 Kafak로 전송(바로 하면 I/O 작업 너무 많아짐)
3. Coordiantor가 이를 컨슘해 분 단위 통계 생성, 실패율이 임계치를 넘으면 일정 시간 서킷 오픈
4. 마이데이터 서버는 10초마다 오픈된 기관 목록을 폴링해 로컬 캐싱하고 해당 트래픽 차단

```
사용자가 기관A 데이터 요청
    ↓
로컬 캐시 확인: "기관A 서킷 오픈 중?"
    ↓ YES
Coordinator나 기관A에 요청도 안 하고 즉시 fallback 응답 반환
```

- 위의 4단계는 폴링을 활용해 Coordinator와 마이데이터 역할을 완전히 분리한 것이다.

#### 서킷 재개

- 가장 처음과 마지막에 배포된 서버 2대를 마스터로 지정한다.(**토스는 카나리 배포 방식**이기에 구서버/신서버 각각 1대씩 하기 위해서)

```
배포 전:  [구버전 A, 구버전 B, 구버전 C]  ← 3대

배포 중:  [구버전 A, 구버전 B, 구버전 C,
         신버전 D, 신버전 E, 신버전 F]  ← 일시적으로 6대 (트래픽이 구→신으로 서서히 전환)

배포 후:  [신버전 D, 신버전 E, 신버전 F]  ← 다시 3대

즉, 구버전 A, 신버전 F가 마스터로 지정된다.
```

- 위 두 대만 빈 응답을 줘서 장애 기관으로 계속 호출하게 만든다.(일부러 장애 기관 하나를 뺌→다시 호출하게 하기 위해서)

## 인프라 점유 시간 단축 - WebSocket 도입

- 자산 등록 API는 한 번의 호출에 인증·토큰 발급·자산 목록·상세 정보 등 여러 번 제공기관 호출이 필요하다.
- 응답시간이 제공기관에 자우되며 한 사례는 (토큰 25초 + 목록 25초 + 상세 24초)로 총 77초 걸린 경우도 있다.
- 유저가 이탈해도 다시 돌아오면 자산 확인은 가능했었는데, API가 길게 처리 되니 토스 공용 인프라(네트워크 장비·게이트웨이)의 점유시간 길어졌다. 트래픽이 많기에 가용성 문제로 이어졌다.
- **WebSocket을 도입해서 마이데이터 서버가 API를 얼리 리턴해 공용 리소스를 즉시 해제**하고, 비동기로 작업을 진행하면서 기관이 완료될 때마다 WebSocket으로 클라이언트에 푸시하도록 했다.

→ 이를 통해 완료 현황을 동적으로 보여줄 수 있게 됐고, 리소스 점유 시간이 평균 3300ms→58ms로 약 56배 개선되었다.

## 피크 트래픽 제어 - 배치 호출 분산

- API는 유저 액션에 의한 유저 호출과 7일에 한 번 자산을 갱신하는 배치 호출로 나뉜다.
- 배치는 하루 6시간(1시~7시)에만 가능한데, 시간은 짧고 유저가 많아 트래픽이 몰리는 문제가 발생했다.
- 해결 실마리를 유저 사용 패턴에서 찾았다.
    - 매월 25일 월급날 아침, 증권은 주말보다 평일
- 구현은 Kotlin 코루틴으로 진행했다.

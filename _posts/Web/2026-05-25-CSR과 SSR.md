## 목표

> CSR과 SSR의 개념과 차이점에 대해서 알아봅니다.

---

## SPA

- **Single** Page Application의 약자로 하나의 페이지로 구성된 웹 애플리케이션이다.
- 웹 사이트에서 화면에서 **범위**를 나누어 클릭한 부분만 바뀌도록 할 수 있다.

## MPA

- **Multi** Page Application의 약자이다.
- 서버로부터 새로운 HTML을 받아와서 페이지 **전체**를 렌더링하는 전통적인 방식이다.

---

## CSR / SSR

#### Client Side Rendering

- 렌더링을 **브라우저(클라이언트)**에서 한다.
- 서버는 빈 HTML과 JS번들(UI 그리기 위한 요소)만 내려주고, JS가 실행되면서 렌더링된다.
- React, Vue, Angular가 대표적인 예이다.

#### Server Side Rendering

- 렌더링을 **서버**에서 한다.
- 요청이 오면 서버가 DB를 조회해 완성된 HTML을 제작해서 보내준다.
- Next.js, Django가 대표적인 예이다.

---

#### 동작 과정

```text
// CSR
1. 브라우저가 서버에 콘텐츠를 요청
2. 서버는 빈 HTML을 응답
3. 브라우저가 연결된 JS 링크를 통해 서버로부터 다시 JS 파일 다운
4. JS를 통해 페이지를 제작해 브라우저에 띄움

// SSR
1. 브라우저가 서버에 콘텐츠를 요청
2. 서버는 필요한 데이터를 모두 삽입 후, CSS까지 적용한 HTML과 JS 코드를 응답
3. 브라우저에서 JS 파일 다운 후, HTML과 JS 로직 연결
```

---

#### 장단점

##### CSR

- 동적으로 DOM을 생성하는 시간을 처음에 기다려야 하기에 초기 로딩 속도가 느리다.
- 서버는 빈 HTM을 브라우저에 넘겨주기에 웹 크롤러가 색인할만한 콘텐츠가 없어서 SEO에 불리하다.
- 하지만, 초기 로딩 이후 페이지 일부를 변경할 때는 해당 데이터만 요청하면 되기에 구동 속도는 빠르다.
- 서버 쪽의 부하가 적어 반응속도가 빠르다.


##### SSR

- 위 동작과정의 2번을 보면 데이터가 이미 담긴 HTML을 전해주기에 SEO에 유리하다.
- 이미 구현되어 있는 HTML이 있기에 초기 구동 속도는 CSR보다 빠르고, 사용자에게는 보여진 이후에 백그라운드에서 JS를 다운받는다.(UX Good)
- 장점이지만, 이는 TTV(Time To View), TTI(Time To Interact) 간 시간 간격이 발상한다는 단점이 있다.
- 서버에서 렌더링하기에 서버 쪽 부하가 있다.

---

## Summary

| | SSR | CSR |
|--|-----|-----|
| 서버가 반환 | 완성된 HTML 전체 | JSON 데이터만 |
| 브라우저가 할 일 | HTML 파싱 + 렌더 | 이미 있는 JS로 그리기 |
| 응답 크기 | 크다 | 작다 |

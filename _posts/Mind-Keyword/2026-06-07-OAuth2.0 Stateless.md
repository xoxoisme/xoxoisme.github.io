> "OAuth2.0를 도입했으나, 세션을 사용하지 않고 완전한 Stateless를 지키고 싶었다."

---

## 목표

> 쿠키 저장소를 직접 구현해봅니다.

---

## 문제상황

- OAuth를 도입하고 네이버로 로그인을 실행한 후, 무한 루프가 돌기 시작했다.
- 처음엔 네이버 쪽 오류인가 했는데 재시도를 계속 해봐도 해결되지 않았다.
- 레퍼런스를 찾아보니 OAuth 인증 객체를 내 백엔드 서버가 세션에 저장하지 못하는 상황이었다.
- 현재 JWT를 사용하기에 세션을 따로 생성하지 않도록 `STATELESS` 로 막아두었다.
- 그러다보니, 세션에 저장을 실패하고 네이버로 리다이렉트 된 후, 콜백이 돌아와도 state가 서로 다르다보니 다시 로그인할려고 `/oauth2/authorization/naver` 로 계속 리다이렉트되면서 무한 루프에 빠진 것이었다.

---

## 해결법

#### 1. STATELESS -> IFREQURIED

- 구현이 단순하다. 단순히 설정만 저렇게 바꾸면 된다.
- OAuth 인증 객체를 세션에 잠시 저장할 때만 세션을 잠시 생성해준다.
- 따로 쿠키 저장소를 만들지 않아도 된다.
- 다만, 전체 인증을 JWT 기반 stateless로 가져가는 게 목표였는데, **OAuth 로그인 흐름만 세션을 요구하면 일관성이 깨진다**는 단점이 있다.

#### 2. 세션이 아닌 쿠키를 이용

- 서버의 stateless 일관성을 지킬 수 있다.
- 또한, 수평 확장 시 세션 동기화가 필요없다.
- **확장할 때 외부 의존성이 없다**는 장점이 너무 크다.
  - 다만, 세션을 사용하고 Redis로 서버 동기화 하는 방법도 있지만 이 또한 네트워크 반복하면서 비용이 낭비된다는 단점이 존재한다.
- 그래서 쿠키를 이용하기로 결정했다.

---

## 해결 과정

1. 먼저 사용자가 로그인 요청을 하는 건 똑같다.

```text
브라우저 → GET /oauth2/authorization/naver
```

2. 커스텀 쿠키 저장소를 따로 만든다.

```text
Spring이 OAuth2AuthorizationRequest 생성
  - state = "a3f9b2c1" (랜덤)
  - redirectUri = ".../login/oauth2/code/naver"
  - clientId, scope 등

         ↓ saveAuthorizationRequest() 호출

직렬화 → Base64 인코딩 → 쿠키 set
Set-Cookie: oauth2_auth_request=eyJzdGF0ZSI...; HttpOnly; Max-Age=180
```

- `saveAuthorizationRequest()` 에서 백엔드가 만든 속성(stat-자동 랜덤 생성, rediretUrl, clientId)을 쿠키에 담는다. 쿠키는 문자열만 저장할 수 있어 객체를 직렬화(바이트 배열로)하여 브라우저의 쿠키에 저장한다.

##### 즉, "Spring이 OAuth 인증 객체를 자동으로 저장하는 걸 인터페이스 따와서 객체를 직접 저장한겠다"는 뜻이다.

- 공격자가 중간에 콜백을 위조하는 걸 막기 위해 해당 쿠키 저장소에서 state를 저장하는 것이다.

3. 이후, Spring이 **자동으로 인가/리소스 서버(콜백 후, 인가 서버는 쿠키로 검증)** 처리 후 DB 저장하고 JWT 토큰을 발급받는다.

```text
콜백 도착 (/login/oauth2/code/naver?code=...&state=a3f9b2c1)
    ↓
[내 쿠키 클래스] loadAuthorizationRequest() → 쿠키에서 저장해둔 state 꺼냄
    ↓
[자동] 네이버가 보낸 state == 쿠키의 state 비교  ← CSRF 검증이 "여기서" 일어남
    ↓ (일치해야만 다음으로)
콜백 도착
    ↓
[자동] 인가 서버 → code로 Access Token 교환
    ↓
[자동] 리소스 서버 → Access Token으로 네이버 사용자 정보 조회
    ↓
[내가 구현] CustomOAuth2UserService.loadUser()
           네이버 정보로 DB 저장/조회
    ↓
[내가 구현] OAuth2SuccessHandler.onAuthenticationSuccess()
           JWT 발급 + 프론트 리다이렉트
```

## 배운 점

- **OAuth2.0의 흐름**을 정확히 파악했다.
- Spring이 OAuth의 state를 원래 어디에 저장하는지 알게되었다. 단순히 프레임워크를 사용만 하는 게 아니라 **분석하는 경험**을 할 수 있었다.
- ‘서버를 완전히 stateless’로 해야겠다는 제약으로 **아키텍처의 일관성에 대해서 다시 고민하는 계기**가 되었다.
- 세션과 쿠키의 **트레이드 오프**를 알게 되었다.
- 직접 구현해보면서, 객체 직렬화/역직렬화, CSRF, XSS(HttpOnly)를 다시 한 번 공부할 수 있었다.

## 인증(Authentication)

- **사용자가 누구인지 확인**하는 과정이다.
- 사용자의 정보가 실제 DB의 정보와 일치한지 확인해서 사용자가 등록되어 있는지 체크하여 등록된 사용자만 접근하도록 하는 접근제어 방식이다.

## 인가(Authorization)

- **사용자가 접근할 수 있는 권리를 가지고 있는지 확인**하는 과정이다.
- 사용자가 요청한 페이지로 갈 수 있는지 권한에 대해서 체크를 하는 제어방식이다.

#### 인가는 인증이 확인된 사용자에게만 적용한다.

## XSS(Cross-Site Scripting)

- 웹 사이트에 **악성 스크립트를 삽입해서** 다른 사용자의 브라우저에서 실행되게 하는 공격이다
- 출력 시 escape, 입력값 검증, `HttpOnly Cookie`, CSP(허용된 스크립트만 실행하도록 제한), React 프레임워크 사용(자동으로 HTML escape)로 방어할 수 있다.

## CSRF(Cross-Site Request Forgery)

- 사용자가 **로그인된 상태를 이용해 원치 않는 요청을 보내게 만드는 공격**이다.
- 서버에 저장된 `sessionId`를 탈취해서 악성 코드가 담긴 서버는 이를 이용해 사용자의 정보를 탈취한다. 서버는 `sessionId` 요청으로 착각해 정보를 내어주는 것이다.
- SameSite Cookie(다른 사이트의 쿠키 전송을 막음), Origin 체크, Spring Security(자동으로 CSRF 방어 기능이 활성화 되어 있음)로 방어할 수 있다.

## Spring security

- Spring에서 제공하는 프레임워크로 다양한 필터체인을 기반으로 인증, 인가, 보안 기능들을 제공한다.

### Filter

- Filter Chain은 `HTTP` 요청을 처리해준다.
- 인증, 권한, 세션 필터 순서로 실행되며 각 필터는 보안 작업을 수행하고 다음의 필터로 요청을 전달한다.

#### Filter Chain은 여기서 DispatcherServlet 앞에서 동작해 request 요청을 처리하기 전에 Filter Chain을 적용할 수 있게 된다.

- Spring Security에서는 보통 `SecurityConfig`에서 `SecurityFilterChain`을 `@Bean`으로 등록해서 사용한다.

### SecurityContext

- 로그인된 사용자 정보를 해당 저장소에 저장해두고 필요할 때 꺼내서 사용한다.

### Principal

- 이는 **현재 로그인한 사용자**를 가리킨다.
- 예를 들어 로그인 사용자가 `username = taehyun, role = USER` 이라고 하자.

```
Principal
  └ UserDetails
       ├ username
       ├ password
       └ roles
```

## JWT 구조

- JWT 토큰은 서명을 만들고 검증할 때 암호화 알고리즘을 사용한다.

```
HEADER.PAYLOAD.SIGNATURE
```

### HEADER

- **토큰의 메타정보**이다.

```json
{
  "alg": "HS256", // 서명 알고리즘
  "typ": "JWT"		// 토큰 타입
}

//Base64Url 인코딩
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
```

### PAYLOAD

- **토큰에 담을 데이터**이다.

```json
{
  "userId": 1,
  "role": "USER",
  "exp": 1710000000
}

//Base64Url 인코딩
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6InRhZWh5dW4iLCJpYXQiOjE1MTYyMzkwMjJ9
```

| Claim | 의미      |
| ----- | --------- |
| sub   | 사용자 ID |
| iat   | 발급 시간 |
| exp   | 만료 시간 |
| iss   | 발급자    |
| role  | 권한      |

- 상호 운용 가능한 클레임 제공을 위해 권장되는 미리 정의된 클레임 집합이다.
- 이는 누구나 읽을 수 있기에 **비밀번호 같은 민감한 정보는 입력하면 안된다.**

### SIGNATURE

- JWT의 **핵심 보안 부분**이다.

```
HMACSHA256(
    base64UrlEncode(header) + "." + base64UrlEncode(payload),
    secretKey
)

// secretKey로 HMACSHA256
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

### Base64Url로 인코딩하는 이유는 JSON 데이터를 `HTTP` 전송에 안전한 문자열 형태로 만들기 위해 한다.

- secretKey만 암호화를 진행하는 이유는 암호화는 **CPU비용이 많이 들고, HEADER와 PAYLOAD는 민감한 정보를 가지고 않기 때문이다.**
